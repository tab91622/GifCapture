# GifCapture 架构与数据流

本文档用于固化模块职责边界，避免后续演进时职责漂移。

## 模块职责

- `src/main.rs`
  - 仅负责应用入口与窗口级配置。
  - 不承载业务状态与录制逻辑。
- `src/app.rs`
  - 负责 UI、状态机、用户输入校验、异步任务编排。
  - 负责录制节奏控制、导出状态切换与结果反馈。
- `src/capture.rs`
  - 负责按区域抓帧并统一返回 `CapturedFrame`。
  - 屏蔽平台细节（当前为 macOS `screencapture`）。
- `src/encoder.rs`
  - 负责将帧序列编码为 GIF。
  - 不关心 UI，仅返回编码成功/失败结果与帧所有权。
- `src/types.rs`
  - 存放跨模块共享数据结构。

## 数据流

1. `app` 根据 `fps` 订阅 `Tick`。
2. 每个 `Tick` 在“单飞保护”下触发一次 `capture::capture_region`。
3. 抓帧成功后回到 `app`，追加到 `frames` 缓冲。
4. 用户点击导出后，`app` 将 `frames` 所有权转给 `encoder`。
5. `encoder` 完成编码后将结果和 `frames` 返还 `app`，用于状态展示与二次导出。

## 关键约束

- 录制态下禁止并发抓帧：上一帧未完成时会跳过当前 `Tick`。
- 导出阶段避免 `frames.clone()` 造成整包内存峰值翻倍。
- `types` 中的结构体保持轻量、可克隆，便于跨异步边界传递。
